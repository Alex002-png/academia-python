# Investigación Pedagógica de N2.M1 — El Servicio

*Construida bajo DOC-11 (T1, Pyodide) y DOC-12 (T2-T6, modalidad `real` — decisión EVT-045). Investigación verificada con WebFetch real.*

## T1 · Decoradores y closures

**Definición oficial (Python Glossary, `docs.python.org/3/glossary.html#term-decorator`, verificado):** *"A function returning another function, usually applied as a function transformation using the `@wrapper` syntax."* La propia documentación oficial muestra la equivalencia semántica exacta que este tema debe enseñar primero: `f = staticmethod(f)` es literalmente lo mismo que `@staticmethod` sobre `def f`. **La sintaxis del decorador es azúcar sintáctico** — esta es la cita que ancla el modelo mental de "caja que envuelve sin abrir": no hay magia nueva, hay una función que recibe otra función y devuelve una tercera.

**Historia real verificada (PEP 318, `peps.python.org/pep-0318`, verificado por fetch directo):** antes de Python 2.4 (2004), aplicar una transformación a una función exigía sintaxis incómoda y repetitiva (`foo = classmethod(foo)`, escrita DESPUÉS de la definición, separando la transformación de la declaración). La sintaxis `@` se eligió por dos razones documentadas: (1) el carácter `@` no se usaba antes en Python, evitando ambigüedad con código previo; (2) ya existía precedente en las anotaciones de Java 1.5. **Hallazgo genuino, con cita textual de Guido van Rossum en el propio PEP:** *"@ is still a fairly arbitrary choice. Some have suggested using | instead."* — la elección del símbolo fue reconocida como arbitraria por su propio creador, no una necesidad matemática. Alternativas como `|decorator` y `[decorator]` se debatieron entre 2002 y 2004 antes de fijarse `@` en Python 2.4a2.

**El puente real hacia FastAPI (`fastapi.tiangolo.com/tutorial/first-steps/`, verificado por fetch directo):** la documentación oficial de FastAPI explica `@app.get("/")` en los mismos términos que este tema debe usar: *"A 'decorator' takes the function below and does something with it... this decorator tells **FastAPI** that the function below corresponds to the path `/` with an operation `get`."* No es una metáfora de esta Academia — es literalmente cómo el framework real describe su propio mecanismo central. **Hallazgo adicional relevante para la arquitectura de N2 (ver `docs/investigacion/n2-arquitectura-servicio-real.md`):** el primer ejemplo del tutorial oficial de FastAPI ya usa `async def root()`, no `def root()` — enseñar `async def` desde el primer contacto con decoradores no es una desviación de la práctica real por una limitación de la plataforma: coincide con el propio ejemplo canónico de la documentación oficial.

## T2 · El primer servicio (FastAPI real, modalidad `real` — EVT-045)

**El comando real verificado (`fastapi.tiangolo.com/#run-it`, fetch directo, oct. 2025+):** la CLI oficial actual es `fastapi dev`, no solo `uvicorn main:app --reload` (que sigue funcionando por debajo — el propio log de salida lo confirma: `INFO: Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)`). La cita **"Press CTRL+C to quit"** es, textualmente, la prueba de que el proceso no termina solo — ancla perfecta para el modelo mental de "ventanilla que nunca cierra" ya declarado en SYL-N2 M1.T2.

**TestClient (`fastapi.tiangolo.com/tutorial/testing/`, fetch directo, verificado):** documentación oficial, cita textual: *"TestClient... is based on HTTPX, which in turn is designed based on Requests"*. El ejemplo canónico oficial usa exactamente el patrón que este tema enseña: `client = TestClient(app)`, `assert response.status_code == 200`. **Nota de arquitectura importante (no aplica aquí, sí en Pyodide):** este ejemplo oficial funciona sin problema en la terminal real del estudiante (hilos de sistema operativo reales disponibles) — la restricción de `async def`/`httpx.AsyncClient` documentada en `docs/investigacion/n2-arquitectura-servicio-real.md` es exclusiva del sandbox Pyodide del Campus y **no aplica a los laboratorios en modalidad `real`**, donde TestClient funciona exactamente como en cualquier máquina de desarrollo real.

## T3 · Contratos con Pydantic

**Definición oficial de validación (`pydantic.dev/docs/validation/latest/concepts/models/`, fetch directo, verificado):** cita textual — *"Initialization of the object will perform all parsing and validation. If no `ValidationError` exception is raised, you know the resulting model instance is valid."* Esta es la base exacta del modelo mental de "doble aduana": la validación ocurre al construir el objeto, no como paso aparte. **`model_dump()`** confirmado como el método oficial de serialización ("Returns a dictionary of the model's fields and values").

**Decisión de diseño:** el contraste ENTRADA/SALIDA (dos modelos Pydantic distintos para el mismo recurso, uno de los cuales oculta un campo interno) no depende de una fuente externa — es una consecuencia directa y verificable del propio mecanismo `response_model` de FastAPI, ya consistente con la explicación oficial de Pydantic sobre validación. Se decidió no buscar un paper de CS-Ed sobre este punto específico (mismo criterio que T1: la mecánica se ancla en documentación oficial primaria, no en literatura de errores de estudiantes).

## T4 · Diseño de una API REST (el Momento Fundacional de N2)

**Fuente primaria verificada (`ics.uci.edu/~fielding/pubs/dissertation/rest_arch_style.htm`, tesis doctoral de Roy Fielding, UC Irvine, 2000, fetch directo):** cita textual — *"REST components perform actions on a resource by using a representation to capture the current or intended state of that resource and transferring that representation between components."* Confirma el ancla histórica ya usada en el diseño de SYL-N2 T4: REST no fue una moda impuesta, fue un patrón **nombrado** a partir de observar qué ya hacía escalar a la web real, en el año 2000.

**Decisión de diseño pedagógico — Bitácora como script real, no solo narrativa verbal:** para que el Momento Fundacional se sienta, no se narre, el laboratorio provee `bitacora_cliente.py`, un script real que el estudiante ejecuta con su propia mano contra su propio servicio, ANTES y DESPUÉS de romper el contrato — el estudiante ve el `KeyError` real en su propia terminal al intentar leer el campo renombrado, no una descripción de que "algo se rompería". Esto sigue el mismo principio ya validado en N1 (Laboratorio 12, DOC-12 §errorVivo): el error inducido en vivo enseña más que la advertencia teórica.

## T5 · Errores como parte del contrato

**Fuente oficial verificada (`fastapi.tiangolo.com/tutorial/handling-errors/`, fetch directo):** `HTTPException` es, en palabras de la propia documentación, *"a normal Python exception with additional data relevant for APIs"*, que se lanza con `raise` (nunca `return`) y detiene el resto de la función inmediatamente. Se reutiliza la taxonomía 4xx/5xx ya verificada en `docs/investigacion/n1-m6-redes-apis.md` (MDN) — sin necesidad de una nueva fuente, honrando DP-07 (fuente única de verdad).

**Decisión pedagógica — la negociación con Bitácora es una decisión, no un ejercicio de código:** SYL-N2 T5 exige explícitamente que el estudiante pueda "decir que no". El laboratorio lo materializa como una conversación real con el tutor (voz de Bitácora) pidiendo un cambio que comprometería el contrato de Laboratorio 14 — el estudiante debe fundamentar el rechazo con un error diseñado (código + mensaje), no solo con una decisión verbal.

## T6 · El contrato se documenta a sí mismo

**Fuente oficial verificada (`fastapi.tiangolo.com/tutorial/metadata/`, fetch directo):** confirmado que los docstrings de una ruta se vuelven su descripción en `/docs` automáticamente, que `summary`/`description` son parámetros explícitos del decorador de ruta, y que toda la documentación nace del propio código vía OpenAPI — nunca se escribe dos veces. Esto ancla directamente el modelo mental "eco de N1.M4.T4" ya declarado en la ficha de SYL-N2 (el README que un tercero sigue sin preguntar).

## Clasificación y falsabilidad

**Evidencia sólida:** las tres citas (Python Glossary, PEP 318, FastAPI docs) verificadas por fetch directo a la fuente primaria/oficial, no por búsqueda ni memoria. **Ausencia declarada:** no se buscó investigación de CS-Ed (SIGCSE/ICER) específica sobre el aprendizaje de decoradores — el tema tiene menos literatura de errores de estudiantes documentada que SQL o HTTP; se construye sobre la propia documentación oficial y la continuidad directa con N1.M3.T3 (`sorted(key=)`), no sobre un misconception externo verificado. **Falsabilidad:** si el registro de ejecución muestra que el puente decorador→`@app.get` no reduce la sensación de "magia" del framework en la práctica, se reconsideraría el énfasis relativo entre la mecánica pura de decoradores y su aplicación inmediata a FastAPI.
