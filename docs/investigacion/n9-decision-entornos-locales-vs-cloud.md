# N9 — Decisión de entornos: local/gratuito vs. cloud real, módulo por módulo

*"Primer paso concreto" de `docs/mision-n9.md`. Verificación empírica previa vía WebSearch (2026-07-21) antes de comprometer el diseño — mismo principio de guía maestra §6/§9: nunca asumir, verificar. Esta decisión determina la ficha de control (DOC-12 §1: Entorno(s) objetivo, prerrequisitos, advertencias) de cada laboratorio que se escriba después.*

## Hallazgo técnico que condiciona el diseño de M1+M2

**Verificado por búsqueda dirigida, no supuesto:**
- vLLM no tiene soporte oficial de Windows, pero **funciona de forma fiable vía WSL2** — el passthrough de GPU es automático con el driver CUDA moderno de WSL, y Ollama/llama.cpp corren al 90-100% de la velocidad nativa de Linux. Puntos de fricción reales documentados a verificar en el primer laboratorio: memoria asignada en `.wslconfig` (el default es solo 50% de la RAM del sistema), instalar `cuda-toolkit-13-x` específico (no el meta-paquete `cuda`, que rompe el stub del driver), y mantener los archivos de modelo dentro del filesystem de WSL (no en `/mnt/c`, 3-5× más lento). *(Fuentes: [vLLM Installation docs](https://docs.vllm.ai/en/latest/getting_started/installation/), [InsiderLLM WSL2 GPU guide 2026](https://insiderllm.com/guides/wsl2-local-ai-windows-guide/))*
- **GPU dentro de Kubernetes es harina de otro costal.** Docker Desktop expone la GPU a contenedores sueltos vía WSL2 sin fricción, pero exponerla como recurso planificable **dentro de un clúster K8s** (incluido `kind`) requiere el NVIDIA device plugin/GPU Operator configurados correctamente — y hay fallos de detección de GPU documentados en 2026 específicamente en esta combinación (Docker Desktop + WSL2 + Kubernetes). *(Fuentes: [Docker GPU support docs](https://docs.docker.com/desktop/features/gpu/), [Lune.dev — Kubernetes no detecta GPU NVIDIA en Docker Desktop+WSL2](https://www.lune.dev/questions/4771/why-isnt-kubernetes-detecting-my-nvidia-gpu-on-docker-desktop-with-wsl))*
- `kind` sí soporta clústeres multi-nodo reales sobre un único host Docker sin ninguna fricción — cada "nodo" es un contenedor, arranque en menos de un minuto. *(Fuente: [kind — Configuration](https://kind.sigs.k8s.io/docs/user/configuration/))*

**Decisión de diseño derivada, con su justificación explícita (no una simplificación oculta):** M1 (Kubernetes) y M2 (vLLM) se mantienen **desacoplados a nivel de GPU**, no por limitación pedagógica sino porque así se construye en la práctica real: la planificación de GPU en Kubernetes es un subdominio especializado (por eso existe el NVIDIA GPU Operator como proyecto separado y maduro) — muchas plataformas de producción reales separan el nivel de aplicación orquestado por K8s del nivel de inferencia servido en nodos GPU dedicados, comunicados por red. M2 sirve el modelo real con vLLM directamente en WSL2 (el camino soportado y verificado), expuesto por su servidor HTTP compatible con OpenAI; M1 orquesta en `kind` la parte de la Columna Vertebral que no depende de GPU (los servicios de aplicación, ingesta, consumidores de eventos) y, cuando corresponda, **llama al servidor vLLM como servicio externo** — arquitectónicamente correcto según el propio DOC-09 (L2 `VLLMProvider` es un adaptador; nada obliga a que el proceso que lo implementa viva dentro del clúster). El GPU Operator queda documentado como **extensión Beyond the Curriculum**, explícitamente fuera de alcance obligatorio, con la razón técnica citada — nunca silenciado como si no existiera.

## Reparto módulo por módulo

| Módulo | Entorno principal | Qué es 100% local/gratis | Qué se declara fuera de alcance (y por qué) |
|---|---|---|---|
| **M1 · Orquestación a escala (K8s)** | `kind`, multi-nodo, en tu máquina | Deployments, Services, ConfigMaps/Secrets, PVs con `local-path-provisioner`, HPA (autoescalado horizontal), rolling updates, todo el API real de K8s | Nodos físicamente separados (kind los simula como contenedores, no como máquinas reales — se declara la diferencia explícitamente); servicios gestionados de un proveedor (EKS/GKE) — se mencionan conceptualmente, no se provisionan; planificación de GPU en K8s (ver hallazgo arriba) — GPU Operator como Beyond the Curriculum |
| **M2 · Model serving (vLLM, Ray)** | vLLM nativo en WSL2, sobre tu RTX 5070 | Servir un modelo real cuantizado (7B-14B) con vLLM; medir latencia/throughput real bajo carga con locust/vegeta; comparar contra un adaptador ingenuo (HF pipeline directo) — antes/después con números propios; Ray Serve local con réplicas múltiples en un solo host para el concepto de autoescalado de réplicas | Tensor/pipeline parallelism multi-GPU (tienes 1 GPU) — se enseña el concepto y se lee la documentación oficial, sin ejecución real; clúster Ray multi-nodo real — mencionado, no ejecutado |
| **M3 · Datos en movimiento (colas, Kafka)** | Redpanda en Docker (protocolo Kafka real) | Productores/consumidores, particiones, grupos de consumo, replay, semántica at-least-once, orden por partición; clúster de 3 brokers Redpanda en Docker Compose para replicación/elección de líder | Kafka gestionado (Confluent Cloud, AWS MSK) — mencionado como equivalente de producción, no provisionado |
| **M4 · Observabilidad y escalabilidad** | Prometheus + Grafana + Loki en Docker, sobre M1/M2/M3 reales | Métricas/logs/alertas reales de tu propio clúster y tu propio servidor vLLM; **trade-offs de coste**: se resuelve con un modelo de costeo honesto — mediciones reales de uso de recursos (CPU/RAM/GPU-hora) de tus propios laboratorios, extrapoladas contra tablas de precios públicas y vigentes de AWS/GCP (un ejercicio real de FinOps: "right-sizing" a partir de utilización medida, nunca gasto real) | Alertas enviadas a un servicio externo real (PagerDuty/Slack) con cuenta de producción — se simula el flujo, no se contrata el servicio |

## Consecuencia para la ficha de control DOC-12 §1 de cada laboratorio

Cada laboratorio de M1-M4 declarará **"Entorno(s) objetivo"** distinguiendo: terminal local (WSL2 para vLLM) vs. contenedor (`kind`/Docker Compose) — nunca "tu sistema operativo" a secas, conforme al propio criterio generalizado de DOC-12 §1. El primer laboratorio del nivel (M1.T1) incluye, como parte de sus **Requisitos verificables** (DOC-12 §2.3), la validación de WSL2 + Docker Desktop + `nvidia-smi` dentro de WSL2 mostrando la RTX 5070 — antes de construir nada encima.
