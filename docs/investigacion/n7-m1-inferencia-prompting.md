## 1. Cómo enseñan este concepto exacto

### (a) Fundamentos de la API — requests, roles, parámetros de sampling

**OpenAI.** La documentación oficial vive ahora en `developers.openai.com` (migrada desde `platform.openai.com/docs`, confirmado por múltiples resultados de búsqueda independientes el 2026-07-21 — hallazgo relevante en sí mismo: cualquier lección que enlace la URL antigua debe verificar el redirect). La Chat Completions API documenta los roles `system`/`developer`, `user`, `assistant` y `tool`: el rol *system* (o su equivalente *developer* en flujos más nuevos) fija personalidad y restricciones y se procesa con peso especial; *user* son las entradas humanas; *assistant* son turnos previos del modelo, necesarios para construir conversación multi-turno. Fuente: [Create chat completion — OpenAI API Reference](https://developers.openai.com/api/reference/resources/chat/subresources/completions/methods/create) (confirmado vía búsqueda, contenido no fetcheado línea por línea — ver nota de honestidad más abajo).

**Anthropic.** La Messages API (`platform.claude.com/docs/en/api/messages`, fetcheada íntegra) declara explícitamente algo que el estudiante que ya conoció OpenAI dará por sentado y que es exactamente incorrecto en Claude: **no existe rol `"system"` dentro del array `messages`**. El prompt de sistema es un parámetro de nivel superior (`system: string` o array de bloques de texto), y el array `messages` solo admite `"user"` y `"assistant"`, alternados (turnos consecutivos del mismo rol se combinan en uno). Cita literal de la documentación: *"there is no `"system"` role for input messages in the Messages API"*. `model`, `messages` y `max_tokens` son obligatorios — `max_tokens` no tiene valor por defecto y debe declararse siempre, a diferencia de otros proveedores. Fuente: [Messages API — Claude Platform Docs](https://platform.claude.com/docs/en/api/messages).

Los parámetros de sampling documentados por Anthropic son `temperature` (rango 0.0–1.0, default 1.0; hacia 0.0 para tareas analíticas, hacia 1.0 para tareas creativas), `top_p` y `top_k` ("recomendados solo para casos de uso avanzados" — la propia documentación desaconseja tocarlos por defecto, dato pedagógicamente útil: evita que el estudiante crea que debe ajustar los tres siempre).

**DeepLearning.AI.** El curso fundacional para este sub-tema es **"ChatGPT Prompt Engineering for Developers"** (Isa Fulford, OpenAI, y Andrew Ng, DeepLearning.AI; 1h40min; 9 lecciones en video, 7 ejemplos de código en notebooks Jupyter). Cubre cómo funcionan los LLM vía API, dos principios de prompting efectivo, y aplicaciones prácticas (resumen, clasificación de sentimiento, extracción de temas, traducción). Fuente verificada: [deeplearning.ai/short-courses/chatgpt-prompt-engineering-for-developers](https://www.deeplearning.ai/short-courses/chatgpt-prompt-engineering-for-developers/).

Un segundo curso, **"Building Systems with the ChatGPT API"** (mismos instructores; 1h55min; 11 lecciones, 9 ejemplos de código, 1 tarea calificada con membresía PRO), extiende esto a "chat formats" (la estructura de roles en una conversación multi-turno), clasificación de queries, moderación de contenido, y encadenamiento de prompts hasta construir un chatbot de atención al cliente. Fuente verificada: [deeplearning.ai/short-courses/building-systems-with-chatgpt](https://www.deeplearning.ai/short-courses/building-systems-with-chatgpt/).

**Honestidad metodológica:** no se buscó ni encontró contenido específico de Hugging Face sobre "fundamentos de la Chat Completions/Messages API de un proveedor comercial" — es un vacío esperable, porque la documentación de Hugging Face gobierna modelos open-weight y su propia API de Inference, no los contratos de OpenAI/Anthropic. ### (b) Streaming de respuestas (SSE)

**Anthropic** (`platform.claude.com/docs/en/build-with-claude/streaming`, fetcheada íntegra) documenta el flujo de eventos SSE con precisión operativa: `message_start` (con `content` vacío) → una serie de bloques de contenido, cada uno con `content_block_start` / uno o más `content_block_delta` / `content_block_stop` → uno o más `message_delta` → `message_stop`. Pueden intercalarse eventos `ping`. Cada `content_block_delta` lleva un `index` que corresponde a su posición en el array `content` final. Existen tres tipos de delta relevantes para M1: `text_delta` (texto), `input_json_delta` (fragmentos parciales de JSON para argumentos de `tool_use` — el modelo solo emite una clave/valor completos por vez, así que el JSON debe acumularse y parsearse recién al `content_block_stop`) y `thinking_delta`.

Dato explícitamente señalado con `<Warning>` en la documentación oficial — candidato directo a error deliberado de Bloque 0/§2.5bis: **el campo `usage` de los eventos `message_delta` es acumulativo**, no incremental — sumar el `usage` de cada `message_delta` produce un conteo de tokens muy superior al real.

Otro hallazgo citable: los errores en streaming (p. ej. `overloaded_error`, HTTP 529 en contexto no-streaming) llegan como un evento `event: error` dentro del propio stream SSE **después** de que la conexión ya devolvió 200 — "no sigue los mecanismos estándar" de manejo de errores HTTP, así que el código de un principiante que solo revisa el status code nunca lo detecta.

**OpenAI** recomienda ahora la Responses API para streaming ("diseñada pensando en streaming... eventos semánticos, type-safe") sobre Chat Completions. Eventos citables: `response.created`, `response.output_text.delta`, `response.completed`, `response.function_call_arguments_delta`, y eventos de error. Fuente: [Streaming API responses — OpenAI API](https://developers.openai.com/api/docs/guides/streaming-responses) (fetcheada). Gotcha documentado explícitamente y poco intuitivo: *"Streaming the model's output in a production application makes it more difficult to moderate the content"* — los scores de moderación de contenido solo están disponibles tras la generación completa, no por delta, así que un pipeline de moderación ingenuo sobre streaming queda ciego mientras el texto llega.

**Errores de novato documentados en foro oficial:** un hilo de la comunidad OpenAI describe streams que se truncan de forma que impide parsear el mensaje final y ver el `finish_reason` real ([Streamed response truncating under token limit](https://community.openai.com/t/streamed-response-truncating-under-token-limit/785925)); y se documenta que en la Responses API, un rate-limit durante streaming se comporta distinto a una llamada normal — el objeto stream se devuelve con éxito y el `APIError` se lanza recién al **leer** el stream, lo que **evita la lógica de reintento automático del cliente** y obliga a implementar retry propio alrededor del bucle de lectura, no de la llamada inicial.

**DeepLearning.AI.** No se encontró ningún curso corto dedicado específicamente a streaming de respuestas de LLM. Se declara la ausencia explícitamente: ni "Building Systems with the ChatGPT API" ni "Serverless LLM Apps with Amazon Bedrock" (curso verificado por fetch directo, hoy retirado del catálogo y redirigido a ofertas más nuevas: transcripción ASR + resumen vía Bedrock/Titan, sin cobertura de streaming, structured output ni gestión de costes) cubren este sub-tema.

### (c) Structured output — JSON mode, function/tool calling

**OpenAI** (`developers.openai.com/api/docs/guides/structured-outputs`, fetcheada) traza una distinción de tres niveles que el estudiante confunde con frecuencia: **JSON mode** solo garantiza JSON válido, sin garantía de esquema; **Structured Outputs** (`strict: true`, disponible desde GPT-4o) garantiza que la salida cumple exactamente el JSON Schema dado — "elimina campos alucinados o valores de enum inválidos"; **function calling** conecta el modelo a herramientas/datos externos, mientras que Structured Outputs da forma a la respuesta final orientada al usuario — la guía oficial es explícita sobre cuándo usar cada una. Ejemplo de código citado con `client.chat.completions.parse(...)` y un modelo Pydantic (`CalendarEvent`).

Gotchas documentados explícitamente por OpenAI mismo: **refusals** (el modelo puede negarse por seguridad y producir una respuesta nula en vez de la estructura esperada — hay que comprobar esto explícitamente, no asumir que siempre llega el esquema), **respuestas incompletas** por alcanzar el límite de tokens (el JSON queda truncado a mitad), y que **solo un subconjunto de JSON Schema está soportado** (restricciones de features). La propia guía recomienda revisar `finish_reason` y manejar refusals de forma explícita en producción.

**Anthropic** (`platform.claude.com/docs/en/agents-and-tools/tool-use/overview`, fetcheada íntegra) distingue **client tools** (ejecutan en la aplicación del estudiante: Claude responde `stop_reason: "tool_use"` con uno o más bloques `tool_use`, el código del estudiante ejecuta la operación real y la devuelve en un bloque `tool_result` en un segundo request) de **server tools** (ejecutan en infraestructura de Anthropic, p. ej. `web_search`). El round-trip completo de un tool client — definir tool con `input_schema` → recibir `tool_use` → ejecutar → responder con `tool_result` referenciando `tool_use_id` → segunda llamada — está documentado con ejemplo end-to-end verificable.

Detalle explícitamente documentado en un `<Accordion>` de la propia guía, con valor pedagógico directo para un error deliberado de laboratorio: **cuando faltan parámetros requeridos en el prompt del usuario, Claude Opus tiende a preguntar por el dato faltante, mientras que Claude Sonnet puede en cambio *inferir* (alucinar) un valor razonable** — p. ej., ante "¿qué tiempo hace?" sin ubicación, Sonnet puede rellenar `"location": "New York, NY"` sin que el usuario lo haya dicho. Esto es un error de novato documentado oficialmente, no una generalización: confiar en que el modelo "siempre pregunta" cuando falta un dato es una asunción que rompe según el modelo elegido.

Para output puramente estructurado (no ligado a tools), Anthropic ofrece un modo separado (`output_format`) que "compila el JSON Schema en una gramática y restringe activamente la generación de tokens durante la inferencia" — no es prompting ("por favor responde en JSON válido"), es una restricción a nivel de decodificación, distinción importante para que el estudiante entienda por qué esto es más confiable que pedirlo en el prompt.

**DeepLearning.AI.** El curso específico es **"Function-calling and data extraction with LLMs"** (Jiantao Jiao, CEO de Nexusflow y profesor en UC Berkeley, y Venkat Srinivasan, Nexusflow; 1h09min; 8 lecciones, 1 quiz, 1 tarea con PRO), verificado por fetch directo. Cubre: crear prompts con definiciones de función y ejecutar las respuestas del LLM, llamadas a múltiples funciones (paralelas y anidadas), uso de especificaciones OpenAPI para acceder a servicios web, y extracción de datos estructurados desde texto no estructurado (incluyendo generación de llamadas SQL desde transcripciones de servicio al cliente). Usa el modelo open-source NexusRavenV2-13B (Hugging Face) en vez de OpenAI/Anthropic — relevante: enseña el *patrón* de function calling de forma agnóstica al proveedor, útil como referencia conceptual, pero el estudiante de N7.M1 debe traducirlo a la sintaxis exacta de OpenAI/Anthropic, que este curso no cubre. Fuente: [deeplearning.ai/short-courses/function-calling-and-data-extraction-with-llms](https://www.deeplearning.ai/short-courses/function-calling-and-data-extraction-with-llms/).

### (d) Gestión de costes — tokens, pricing real, control de gasto

**Precios verificados el 2026-07-21, directo de documentación oficial — nunca de memoria:**

*OpenAI* (`developers.openai.com/api/docs/pricing`, fetcheada), por millón de tokens: GPT-5.6-Sol $5 input / $30 output; GPT-5.6-Luna $1 / $6; GPT-5.4 $2.50 / $15; GPT-5.4-Nano $0.20 / $1.25. Batch API: 50% de descuento sobre ambos (GPT-5.6-Sol batch: $2.50 / $15). Prompt caching: el "cached input" de GPT-5.6-Sol cuesta $0.50/MTok frente a $5.00 estándar — 90% de descuento — pero **escribir** en la caché tiene coste propio (ejemplo citado: $6.25/MTok para GPT-5.6-Sol), dato que un principiante que solo mira el precio de lectura de caché puede pasar por alto. Recargo de 10% para endpoints con residencia de datos regional en modelos lanzados desde marzo de 2026.

*Anthropic* (`platform.claude.com/docs/en/about-claude/pricing`, fetcheada íntegra — la fuente más detallada de esta investigación), por millón de tokens: Claude Opus 4.8 $5 input / $25 output; Claude Sonnet 5 **$2/$10 en precio introductorio vigente hasta el 31 de agosto de 2026**, después $3/$15 estándar (fecha y cambio de precio documentados explícitamente, dato con vencimiento real que el laboratorio debe declarar con su fecha de verificación); Claude Haiku 4.5 $1/$5. Prompt caching con tres multiplicadores exactos sobre el precio base de input: escritura de caché de 5 minutos ×1.25, escritura de caché de 1 hora ×2, lectura de caché (cache hit) ×0.1 — es decir, un 90% de descuento en cada lectura, con el punto de equilibrio explícito ("la caché se amortiza tras una sola lectura para la ventana de 5 minutos, tras dos lecturas para la de 1 hora"). Batch API: 50% de descuento sobre input y output, combinable con prompt caching. Coste oculto documentado explícitamente y con tabla propia: **usar `tools` añade un system prompt automático de entre ~290 y ~800 tokens según el modelo y el `tool_choice`**, coste añadido que no aparece en el prompt que el estudiante escribió y que solo se ve reflejado en el campo `usage` de la respuesta.

La documentación de pricing de Anthropic incluye un **ejemplo resuelto paso a paso** (una sesión de una hora con Opus 4.8, 50.000 tokens de input y 15.000 de output = $0.705 sin caché, $0.525 con el 80% de esos inputs servidos desde caché) — material directamente reutilizable como ejercicio de cálculo de coste real en el laboratorio de M1.

**Rate limits y sorpresas de coste:** OpenAI mide en RPM/TPM/TPD/RPD/IPM, y advierte explícitamente que "los límites pueden alcanzarse por cualquiera de las métricas, dependiendo de cuál ocurra primero" — el ejemplo citado en la propia guía es que 20 requests de solo 100 tokens cada uno agotan el RPM antes de tocar el TPM. Fuente: [Rate limits — OpenAI API](https://developers.openai.com/api/docs/guides/rate-limits) (fetcheada). Un error de novato explícito de la misma guía: reintentar sin pausa cuenta contra el límite y empeora el problema — el patrón correcto es backoff exponencial con jitter.

Anthropic documenta un matiz que rompe una asunción común: **el rate limit se aplica a nivel de organización, no por API key** — todas las keys de una cuenta comparten el mismo pool, así que un estudiante que cree estar aislado usando una key nueva para "resetear" su límite no lo consigue. Fuente: [Our approach to API rate limits — Anthropic Support](https://support.anthropic.com/en/articles/8243635-our-approach-to-api-rate-limits) (verificada por búsqueda, contenido no fetcheado íntegro — ver nota de honestidad). La documentación oficial de errores (`platform.claude.com/docs/en/api/errors`, fetcheada íntegra) confirma el manejo recomendado: reintento con backoff exponencial, respetando el header `retry-after`; los SDKs oficiales ya reintentan automáticamente (dos veces por defecto) errores transitorios, y distingue explícitamente el caso 529 `overloaded_error` (sobrecarga global de la API, no del estudiante) del 429 `rate_limit_error` (límite de la cuenta), dos causas que un principiante suele tratar como si fueran lo mismo.

Sorpresa de coste no ligada a un solo request sino a un patrón de uso completo, documentada en varias fuentes secundarias sobre rate limiting: **el modelo no procesa solo el mensaje nuevo — recibe todo el historial de la conversación como input en cada llamada**, así que una conversación de 50 turnos reenvía esos 50 turnos como tokens de entrada en el turno 51. Es el motor real detrás de la "sorpresa de factura" más común en apps conversacionales sin gestión de contexto, y conecta directamente con la necesidad de truncar o resumir historial — tema que un laboratorio de gestión de costes de N7.M1 debe cubrir explícitamente.

**DeepLearning.AI.** No se encontró ningún curso corto dedicado específicamente a gestión de costes/tokens/pricing de APIs de LLM. Se declara la ausencia explícitamente en vez de forzar una cita — es coherente con que DeepLearning.AI enseña principalmente construcción de aplicaciones, no operación/FinOps de las mismas.

## 3. Síntesis crítica

**Qué mecanismo específico funciona y por qué — nunca una colección de "OpenAI hace X, Anthropic hace Y".**

1. **La asimetría de roles entre proveedores es, en sí misma, el mejor conflicto cognitivo disponible para (a).** No hace falta inventar un error deliberado: la propia divergencia real entre "system es un rol de mensaje" (OpenAI) y "system es un parámetro top-level, no hay rol system" (Anthropic, declarado explícitamente en su documentación oficial con esa frase exacta) es un conflicto cognitivo auténtico, verificable y con fuente primaria — exactamente el tipo de material que DOC-03 exige para el Bloque 2 (Conflicto cognitivo): el estudiante que aprendió primero con un proveedor construye un modelo mental incorrecto que se rompe visiblemente al cambiar de proveedor, no por un bug artificial sino por una diferencia real de diseño de API. Esto es superior a inventar un bug de sintaxis porque enseña algo que el estudiante necesitará el día que trabaje con el segundo proveedor.

2. **Streaming se enseña mejor mostrando primero la respuesta "de golpe" (no streaming) y después el mismo request con `stream: true`, no explicando SSE en abstracto.** La documentación oficial de ambos proveedores sigue este patrón — muestra primero `messages.create()`/`chat.completions.create()` sin streaming, y solo entonces introduce el mismo objeto de respuesta reconstruido evento a evento. Esto es coherente con el principio ya institucionalizado en DOC-12 §0.1 ("concepto antes que comando/sintaxis") aplicado aquí a nivel de request: el estudiante primero entiende *qué* estructura de datos recibe (un objeto `Message`/`ChatCompletion` completo), y solo después entiende que streaming es la misma estructura entregada en fragmentos — nunca al revés. El Warning oficial sobre `usage` acumulativo en `message_delta` es el error deliberado ideal para este punto: no es un error de sintaxis, es un error de modelo mental sobre qué representa cada evento, exactamente la clase de bug que DOC-11 Bloque 0 exige (M11: "representa una clase de error, no un typo aislado").

3. **Structured output debe enseñarse como una escalera de tres niveles de garantía, no como tres features paralelas.** La propia documentación de OpenAI traza esta jerarquía explícitamente (texto libre → JSON mode → Structured Outputs con `strict: true`), y Anthropic la refleja con su propia distinción (prompting "por favor responde en JSON" vs. `output_format`/`strict: true` que restringe la decodificación). Enseñar los tres niveles en ese orden, mostrando *por qué* cada nivel resuelve una falla real del anterior (texto libre falla por parseo; JSON mode falla por campos faltantes/alucinados; Structured Outputs lo resuelve por restricción de gramática) es más fuerte pedagógicamente que presentar "aquí está la feature" sin la falla que la motivó — mismo principio C1 de DOC-03 ("el estudiante nunca aprende una herramienta, aprende la respuesta histórica a un problema"), aplicado aquí a nivel de evolución de una única API en vez de historia de una tecnología completa.

4. **La gestión de costes es el único sub-tema de M1 donde la fuente más fuerte no es un curso ni documentación conceptual, sino una tabla de precios con fecha de caducidad real.** Esto es una restricción de diseño, no una elección: el precio introductorio de Sonnet 5 vence el 31 de agosto de 2026, y los precios de tokens cambian con cada nuevo modelo. Un laboratorio de N7.M1 que hardcodee cifras de precio en el texto de la lección queda obsoleto por diseño. La decisión correcta — y esto sí es una decisión de diseño explícita de esta investigación, no un hallazgo externo — es que el laboratorio enseñe **el método de cálculo** (tokens × tarifa vigente, con y sin caché/batch) usando el ejemplo resuelto que la propia documentación de Anthropic ya publica (sesión de 1h, 50k/15k tokens, $0.705 → $0.525 con caché), y remita al estudiante a comprobar la tarifa vigente en la página oficial en el momento de hacer el laboratorio, en vez de imprimir una cifra que DOC-12 §6 tendría que corregir cada pocos meses.

**Limitación frente a la identidad pedagógica propia (DOC-03):** ninguna de las fuentes revisadas (documentación oficial, cursos de DeepLearning.AI) enseña con conflicto cognitivo explícito, diagnóstico relámpago, ni escalera de seis categorías — son documentación técnica de referencia y cursos de aplicación práctica, no diseño pedagógico en el sentido de DOC-03/DOC-11/DOC-12. Esto no es una carencia de la investigación: es exactamente lo esperable, y confirma que la Academia no debe "copiar su estructura" (mandato explícito de DOC-12 §3) — debe extraer los hechos técnicos verificados y las fallas documentadas, y construir el andamiaje pedagógico (Bloques 0–6 / secciones 2.1–2.13) desde DOC-03/DOC-12, no desde las fuentes.

## 4. Estrategia propuesta — secuenciación de temas para N7.M1

*Clasificación de evidencia de cada decisión, conforme a la revisión crítica adversarial exigida por DOC-11 §0bis (heredada por DOC-12 §0bis): **evidencia sólida** · **evidencia limitada** · **decisión de diseño**.*

**T1 — Fundamentos de la API: requests, roles, parámetros** *(decisión de diseño respaldada por evidencia sólida de la asimetría de roles)*
Primer tema porque todo lo demás (streaming, tools, costes) se construye sobre el objeto de request/response base. Debe cubrir ambos proveedores desde el día 1 (no "primero OpenAI, después Anthropic") precisamente para que la asimetría de roles (system-como-mensaje vs. system-como-parámetro) funcione como conflicto cognitivo real y no como una sorpresa tardía en T2. Recursos: documentación oficial de ambos + "ChatGPT Prompt Engineering for Developers" (DL.AI) para el patrón de prompting iterativo. Profundidad: alta en roles y en `max_tokens`/`temperature` (con su rango real, verificado por proveedor — 0.0–1.0 en Anthropic, distinto en OpenAI); baja en `top_p`/`top_k` (la propia documentación de Anthropic los marca "avanzado").

**T2 — Streaming de respuestas** *(evidencia sólida para el patrón "no-streaming primero, streaming después"; evidencia limitada para el orden streaming-antes-de-tools frente a la alternativa)*
Segundo porque depende de T1 (el estudiante ya sabe qué es un `Message`/`ChatCompletion` antes de verlo fragmentado) y porque introduce el objeto que T3 (tool use) reutiliza en su forma streamed (`input_json_delta`). Error deliberado obligatorio (DOC-12 §2.5bis): provocar que el estudiante sume manualmente `usage` de cada `message_delta` y compare contra el `usage` real — reproduce exactamente el Warning oficial. Recursos: streaming docs de ambos proveedores (fetcheadas íntegras en esta investigación); no hay curso DL.AI dedicado — se declara así en el laboratorio.

**T3 — Structured output: JSON mode y tool/function calling** *(evidencia sólida para la escalera de tres niveles de garantía; decisión de diseño para enseñar tool calling con Anthropic primero, ver justificación)*
Tercero porque reutiliza tanto el modelo mental de roles (T1) como el de streaming de bloques (T2, para `input_json_delta`). Se propone enseñar el round-trip de tool use con Anthropic primero (el flujo `tool_use`→ejecutar→`tool_result` es más explícito y menos "mágico" que el equivalente OpenAI) y JSON mode/Structured Outputs de OpenAI después, apoyado en "Function-calling and data extraction with LLMs" (DL.AI) como referencia conceptual agnóstica de proveedor. Error deliberado candidato: el caso documentado de Sonnet alucinando un parámetro faltante en vez de preguntar — mini-laboratorio ideal porque es reproducible y tiene fuente oficial.

**T4 — Gestión de costes: tokens, pricing, control de gasto** *(decisión de diseño explícita — pricing como método, no como cifra)*
Último porque requiere que el estudiante ya haya generado tráfico real con T1–T3 para poder leer su propio `usage` con contexto. Contenido: cómo se cuentan los tokens; tabla de precios vigente **con fecha de verificación explícita en el propio laboratorio** y un puntero a la página oficial para revalidar; el ejemplo resuelto de Anthropic (1h/50k/15k tokens, con y sin caché) adaptado como ejercicio; rate limits (RPM/TPM, backoff exponencial, el matiz de límite por organización no por key); y el enlace directo con T1–T2: coste oculto de tools (system prompt automático) y coste de reenviar historial completo en streaming/multi-turno. Cierra el módulo porque integra los tres temas anteriores en una sola pregunta económica real, coherente con el mandato de mini-proyecto integrador de DOC-12/DOC-11.

**Por qué no un orden alternativo (costes primero):** se consideró y se rechaza — enseñar pricing antes de que el estudiante haya hecho una sola llamada real a la API es abstracto y no verificable por el propio estudiante (no tiene un `usage` real que interpretar). Rechazo justificado con traza concreta (principio 1, DOC-11 §0bis): sin T1–T3 previos, el ejercicio de cálculo de coste del ejemplo de Anthropic sería un ejercicio de aritmética sin conexión con la propia experiencia del estudiante, exactamente el modo de fallo que DOC-03 P3 identifica como inefectivo.

## Decisiones de diseño derivadas de la investigación

*(Sección de cierre obligatoria, DOC-11 §0bis principio 5, heredada por DOC-12 §0bis.)*

- **Se adopta**: enseñar roles y parámetros de ambos proveedores en paralelo desde T1, usando la asimetría real system-mensaje/system-parámetro como conflicto cognitivo — respaldo: evidencia sólida (documentación oficial explícita de ambos proveedores).
- **Se adopta**: secuencia fundamentos → streaming → structured output → costes, con costes al final — respaldo: decisión de diseño justificada por rechazo explícito de la alternativa (costes primero), no evidencia externa de Learning Sciences (no localizada).
- **Se adopta**: el Warning oficial sobre `usage` acumulativo en streaming como error deliberado obligatorio de T2 — respaldo: evidencia sólida, fuente primaria fetcheada.
- **Se adopta**: el caso documentado de Sonnet alucinando parámetros faltantes como error deliberado candidato de T3 — respaldo: evidencia sólida, fuente primaria fetcheada (Accordion oficial de Anthropic).
- **Se modifica**: no se hardcodea ninguna cifra de precio como verdad permanente del laboratorio — se enseña el método de cálculo con un ejemplo oficial resuelto, y se exige verificación de tarifa vigente en el momento de ejecutar el laboratorio — respaldo: los propios precios documentados incluyen una fecha de vencimiento explícita (Sonnet 5, 31 de agosto de 2026), evidencia directa de que cualquier cifra fija caduca por diseño.
- **Se rechaza**: usar "Serverless LLM Apps with Amazon Bedrock" (DL.AI) como recurso de M1 — motivo trazable: fetch directo confirma que el curso está retirado del catálogo y no cubre streaming, structured output ni gestión de costes; su contenido (ASR + resumen vía Bedrock/Titan) no es distintivo para ninguno de los cuatro sub-temas de este módulo.
- **Pendiente de validación** (nunca "pendiente de desarrollo", regla de nomenclatura DOC-11 §0bis): si el orden Anthropic-primero/OpenAI-después en T3 (tool use) produce mejor transferencia que el orden inverso — hipótesis abierta sin evidencia externa localizada; a resolver con el registro de observaciones de ejecución (§6) cuando el laboratorio se imparta con un estudiante real.
- **Falsabilidad explícita** (principio DOC-11 §0bis): la decisión de secuenciar costes al final se abandonaría si el registro de ejecución (§6) mostrara que los estudiantes llegan a T4 habiendo ya gastado cuota real de forma descontrolada en T1–T3 por no haber visto antes ningún límite — en ese caso, un checkpoint ligero de costes movido a T1 (sin el laboratorio completo) sería la corrección indicada.
